# Plan: Remove Sound ESP, Add Footsteps ESP

## Overview
Remove all Sound ESP code and replace it with a **Footsteps ESP** feature that draws expanding circles at enemy feet positions. Circles appear on footsteps, jumps, and landings, expanding to a radius proportional to the action intensity, then fading out. Circles are parallel to the ground (drawn in 3D world space, projected to 2D screen).

```cpp
struct FootstepRing {
  SDK::Vector3 worldPos;    // Feet position where sound occurred
  float color[4];           // RGBA
  float maxRadius;          // Max radius in world units
  double startTime;         // ImGui::GetTime() when ring was created
  bool isLandRing;          // true = landing (use ground-projected circle)
};

class FootstepsEsp : public IFeature {
public:
  void Update() override;
  void Render(Render::DrawList &drawList) override;
  std::string_view GetName() override { return "FootstepsEsp"; }
  void RenderUI() override;

private:
  std::vector<FootstepRing> m_rings;
  std::unordered_map<uintptr_t, bool> m_prevOnGround;  // entity address -> was on ground
};
```

### New feature impl: `src/features/footsteps_esp/footsteps_esp.cpp`

**Update()** — called each render frame:
1. Get current players from `GameManager::GetRenderPlayers()`
2. For each valid, alive player:
   - Read `m_fFlags` from entity memory: `MemoryManager::Read<uint32_t>(entity.address + Offsets::m_fFlags)`
   - Check bit 0: `bool isOnGround = (flags & 1) != 0`
   - Check previous state from `m_prevOnGround[entity.address]`
   - Read entity velocity: `MemoryManager::Read<Vector3>(entity.address + Offsets::m_vecVelocity)` (need to add this offset — `m_vecVelocity` = `0x140` on `C_BaseEntity`)
   - Detect transitions:
     - **Walking**: `wasOnGround && isOnGround && velocity.magnitude() > 50` → small ring
     - **Jumping**: `wasOnGround && !isOnGround` → medium ring
     - **Landing**: `!wasOnGround && isOnGround` → large ring (radius proportional to velocity magnitude)
   - Create `FootstepRing` with appropriate color and max radius
   - Update `m_prevOnGround[entity.address] = isOnGround`
3. Clean up expired rings (older than `expandDuration + fadeDuration`)

**Render()** — called each frame:
1. If not enabled, return
2. Get `viewMatrix` from `GameManager`
3. Get screen width/height from `Overlay`
4. For each ring in `m_rings`:
   - Calculate elapsed time: `elapsed = now - ring.startTime`
   - If `elapsed > expandDuration + fadeDuration`, skip (expired)
   - Calculate current radius:
     - If `elapsed < expandDuration`: `radius = maxRadius * (elapsed / expandDuration)`
     - Else: `radius = maxRadius`
   - Calculate alpha:
     - If `elapsed < expandDuration`: `alpha = baseAlpha`
     - Else: `alpha = baseAlpha * (1.0 - (elapsed - expandDuration) / fadeDuration)`
   - World-to-screen the ring position
   - Scale radius to screen pixels: `screenRadius = radius * (screenHeight / viewMatrix[1][1] / ringPos.z)` (approximate perspective scaling)
   - Draw circle with `DrawCircle(screenX, screenY, screenRadius, color, segments, thickness)`

## Key Technical Details

### Circle rendering approach
The circles are drawn **in world space** at the entity's feet position, parallel to the ground. Since `DrawCircle` is a 2D screen-space method, we:
1. World-to-screen the entity's feet position
2. Calculate an approximate screen-space radius by projecting a point offset by `radius` world units from the feet position
3. Draw the 2D circle at the screen position with the calculated radius

This gives the visual effect of a 3D circle parallel to the ground, correctly scaled by perspective.

### Ground detection
- `m_fFlags` at `0x400` on `C_BaseEntity` — bit 0 = `FL_ONGROUND`
- `flags & 1` → entity is touching ground
- Track per-entity previous state to detect transitions

### Velocity-based intensity
- `m_vecVelocity` at `0x140` on `C_BaseEntity` — `Vector3` velocity
- Walking: velocity 50-200 units/s → small ring
- Running: velocity 200+ units/s → medium ring
- Landing: velocity magnitude determines ring size (higher fall = bigger ring)

### Threading note
Since `Update()` and `Render()` are called from the render thread, and we're doing `MemoryManager::Read()` calls directly in the feature's `Update()`, this is acceptable — the render thread already does live reads for aimbot/triggerbot. The entity addresses are stable within a single frame.
